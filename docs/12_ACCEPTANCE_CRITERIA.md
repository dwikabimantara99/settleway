# 12 - Acceptance Criteria

## Phase 0

- `web/` app exists.
- App starts locally.
- Root docs remain intact.

## Phase 1

- Landing page exists.
- Reusable components exist.
- App has Settleway branding and role switch.

## Phase 2

- Marketplace route works.
- Listing detail route works.
- Buyer request route works.
- Profile route works.

## Phase 3

- Deal Room route works.
- Money breakdown is correct.
- Status stepper is visible.
- Role-based actions are visible.

## Phase 4

- API routes exist.
- Demo data is served from API or fallback store.
- Database schema file exists.

## Phase 5

- Buyer deposit changes state.
- Seller deposit changes state.
- Both deposits lock escrow.
- Invalid transitions are rejected.

## Phase 6

- Soroban contract builds.
- Contract has tests.
- Events/status transitions are represented.

## Phase 7

- Backend stores Stellar metadata.
- Deal Room shows contract/tx/proof identifiers.
- Fallback mode is explicit.

## Phase 8

- Evidence can be uploaded/simulated.
- Hash is generated.
- Reputation updates after completion.
- Privacy toggle affects public proof display.

## Phase 9

- `/demo` guided flow works.
- Reset demo state works.
- `pnpm lint` and `pnpm build` succeed or failures are documented.
